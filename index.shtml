ABB Nested Cards

Cards are used to display an assembly of related content in self-contained 'packages'.

ABB cards are made up of a number of modular sub-components containers which can be omitted if not needed. Thay can also have more than one side for example and 'front' and a 'back' side.

Anatomy of an ABB card

The following is a description of all the ABB card containers:

  • abbCard-deck
    This is the parent container for one or more cards. The abbCard-deck sets the rules for how the individual cards should behave regarding layout and responsiveness.
  • abbCard-wrapper
    Each card has a wrapper to hold all it's parts together.
  • abbCard-head
    The head is typically used for placing the card title or similar information in it. Other typical elements placed int he head are close buttons (X), dates, icons etc.
  • abbcard-body-front
    The front container hold the parts that are shown as default.
  • abbcard-body-back
    If there is a back side to the card then it is usually toggled via a button placed in the foot of the card.
  • abbCard-core - this is the part that will be interchangeable with other cards and itself. It can be placed inside another abbCard-core-details

    • abbCard-core-displayBox
      This is used to show almost any media type - images, carousels of images, movie clips, text etc.
    • abbCard-core-controls
      The controls container can be used to place a set of buttons controlling the content of the card for example tabs which swap out the content in a nested card core or the content of one of it's siblings such as the details container.
    • abbCard-core-details
  • abbCard-foot
    The foot is usually used to hold a toggle button or navigation controls.

NB! Omitting any of the containers should result in the remaining containers to fill the void. this is to maintain a level of visual consistency.

Dimensions

The size of cards should be small enough to show enough information for a viewer to be able to find enough useful information to help them find what they are looking for. There are no fixed rules for the actual dimensions. Card's content should determine it's dimensions to a degree.

Spacing (M + P)

It is a good idea to allow for some margin space between rows and columns of cards in order to assist understanding of where one card ends and another begins. It is also advisable to use padding within the card itself to pull the content away for the edges.

Responsiveness

Cards should have responsive properties built into them. The card deck can be used to control this feature. Cards themselves should have a min and max width setting so that the card deck can control stacking direction (vertical or horizontal), and for controlling how many cards should be able to placed side-by-side.

So far we have been placing 3 cards into each card deck without much thought about why they resize the way they do (their default width and their minimum width when the window is resized). The reason for this behavious is because of the flexbox properties of the card deck container. It states that each card inside it should equally share the available horizontal space.
